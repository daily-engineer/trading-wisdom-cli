# 贡献指南

欢迎为 Trading Wisdom CLI 项目贡献想法和反馈！

## 📝 如何提出反馈

### 方式 1: GitHub Issues（推荐）

最好的方式是通过 GitHub Issues 提出反馈：

1. 访问 [Issues 页面](https://github.com/daily-engineer/trading-wisdom-cli/issues)
2. 点击 **New Issue**
3. 选择 **📝 文档反馈或建议**
4. 填写模板并提交

**优势**:
- ✅ 公开透明
- ✅ 易于追踪和讨论
- ✅ 自动生成通知
- ✅ 可以附加标签和里程碑

### 方式 2: 直接编辑（如果你是 Collaborator）

如果有 Collaborator 权限，可以直接编辑：

1. 访问 repo 中的 `/docs` 文件夹
2. 选择要编辑的文件
3. 点击编辑图标
4. 提交 Pull Request

### 方式 3: Pull Request（推荐用于代码）

对于代码贡献：

1. Fork 这个 repo
2. 创建一个新分支: `git checkout -b feature/your-feature`
3. 进行修改
4. Commit: `git commit -m "feat: description"`
5. Push: `git push origin feature/your-feature`
6. 在 GitHub 上创建 Pull Request

## 📋 反馈模板

### 文档反馈

```
## 文档名称
Roadmap.md

## 问题描述
[描述你发现的问题或改进建议]

## 改进建议
[如何改进？]

## 相关内容
[引用相关的行或部分]
```

### 设计反馈

```
## 设计模块
[哪个模块/组件]

## 当前设计
[现在的设计是什么]

## 建议改进
[你的改进建议]

## 理由
[为什么这样改进更好]
```

## 🎯 反馈评审流程

1. **提交** - 你提出反馈
2. **讨论** - 团队成员在 Issue 下评论讨论
3. **决策** - 决定是否采纳
4. **实施** - 执行改进
5. **反馈** - 更新你反馈的 Issue

## 📚 文档位置

- **项目计划**: `/docs/Roadmap.md`
- **架构设计**: `/docs/Architecture.md`
- **CLI设计**: `/docs/CLI-Design.md`
- **主页**: `/docs/README.md`

## 🏷️ Issue 标签说明

| 标签 | 含义 |
|------|------|
| `feedback` | 反馈或建议 |
| `documentation` | 文档相关 |
| `architecture` | 架构设计 |
| `planning` | 项目计划 |
| `enhancement` | 功能改进 |
| `bug` | 问题或错误 |

## 💡 提出好反馈的要点

✅ **清晰** - 明确说明问题或建议
✅ **具体** - 提供具体的例子或链接
✅ **构造性** - 提出改进建议
✅ **尊重** - 尊重他人的观点
✅ **相关** - 确保反馈与项目相关

## 🚀 反馈示例

### 好的反馈 ✅

```
## Issue: Roadmap.md 中缺少风险评估

在 Roadmap.md 的 Phase 1 部分，
我注意到没有列出潜在的风险。

建议添加一个 "已知风险" 小节，
包括：
- 外部依赖风险
- 技术可行性风险
- 时间估算偏差风险

这样可以帮助团队更好地准备应对措施。
```

### 不好的反馈 ❌

```
文档写得不好，需要改进。
```
